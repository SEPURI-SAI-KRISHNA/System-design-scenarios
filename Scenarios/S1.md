
### Foundations of Scalability & Data Ingestion

#### **1. EASY: The Daily Log Archiver**

**Scenario:** You work for a startup running a monolithic web application on a single Linux server. The app generates 5GB of text logs daily.
**Task:** Design a system to move these logs to cheaper long-term storage every night at 2 AM for compliance. The system must alert if the transfer fails.

**Solution Architecture:**

* **Source:** Local File System (Linux).
* **Transport:** Python Script triggered by `crontab`.
* **Destination:** AWS S3 (Infrequent Access Tier).
* **Monitoring:** AWS SNS (Simple Notification Service) or Slack Webhook.

**The Workflow:**

1. **Cron Job:** Triggers at 02:00.
2. **Script:** Compresses logs (`gzip`) to reduce network I/O and storage cost.
3. **Upload:** Uses `boto3` to perform a multipart upload to S3.
4. **Verification:** Checks MD5 checksum of the uploaded file against the local file.
5. **Cleanup:** Deletes local logs older than 7 days.
6. **Alerting:** If any step raises an Exception, send a POST request to a Slack Webhook.

**Why this is the best solution:**

* **Simplicity:** Over-engineering this with Kafka or Airflow is unnecessary for a single server.
* **Cost:** S3 Infrequent Access is the cheapest reliable storage for data rarely accessed.
* **Reliability:** Multipart upload handles network blips better than a single stream.

#### **2. MEDIUM: Real-Time Stock Trading Dashboard**

**Scenario:** A fintech startup needs a dashboard that displays stock prices with less than 1-second latency.
**Task:** Build a data pipeline that ingests stock ticks from an external API (NASDAQ/NYSE), processes them (calculates moving averages), and serves them to a frontend via WebSockets.
**Constraints:** High throughput (100k events/sec), low latency.

**Solution Architecture:**

* **Ingestion:** GoLang Producer (High concurrency).
* **Message Broker:** Apache Kafka.
* **Stream Processing:** Apache Flink.
* **Serving Layer:** Redis (Pub/Sub & Cache).
* **Frontend Comms:** Node.js + Socket.io.

**The Workflow:**

1. **Ingestion:** A cluster of Go services connects to the Stock Exchange TCP socket and pushes raw bytes into a Kafka topic (`raw-ticks`) partitioned by `StockSymbol`.
2. **Processing:**  Apache Flink consumes `raw-ticks`. It uses a **Tumbling Window** of 1 second to calculate the "Start, End, High, Low" price.
3. **Storage:** Flink pushes the aggregated result to Redis.
4. **Delivery:** The Node.js WebSocket server subscribes to Redis channels. When Redis updates, Node.js pushes the update immediately to the client browser.

**Why this is the best solution:**

* **Kafka:** Handles the 100k/sec burst without data loss. Partitioning by Symbol ensures ordering.
* **Flink vs. Spark:** Flink is chosen over Spark Structured Streaming because Flink is true row-based streaming (lower latency), whereas Spark is micro-batch (higher latency).
* **Redis:** In-memory speed is non-negotiable for sub-second requirements.

#### **3. HARD: The "Twitter Snowflake" (Unique ID Generator)**

**Scenario:** You are the Lead Architect at a company like Twitter. You need to generate unique IDs for every tweet.
**Task:** Design a distributed ID generator system.
**Constraints:**

1. **Unique:** IDs must be unique across the globe.
2. **Sortable:** IDs must be roughly sortable by time (if ID A > ID B, then A was created later).
3. **Numeric:** Must be 64-bit integers (for database indexing efficiency).
4. **Scale:** Must handle 100,000 requests per second.
5. **Availability:** The system cannot go down.

*Note: You cannot use UUIDs (128-bit, not time-sortable, bad for DB indexing).*

**Solution Architecture:**
We implement a custom **Snowflake ID** algorithm (originally designed by Twitter). We do not use a centralized database (ticket server) because it becomes a write bottleneck.

**The Structure (64 bits total):**


$$\text{Sign bit (1)} | \text{Timestamp (41)} | \text{Data Center ID (5)} | \text{Machine ID (5)} | \text{Sequence (12)}$$

1. **Sign Bit (1 bit):** Unused (always 0) to keep the number positive.
2. **Timestamp (41 bits):** Milliseconds since a custom epoch (e.g., Nov 29, 2025). Gives us 69 years of IDs.
3. **Datacenter ID (5 bits):** Allows 32 data centers.
4. **Machine ID (5 bits):** Allows 32 worker nodes per data center.
5. **Sequence (12 bits):** For every millisecond, a single machine can generate $2^{12} = 4096$ unique IDs.

**The Infrastructure:**

* **Compute:** A cluster of lightweight Java/Scala instances (ZooKeeper is used to assign Machine IDs to these instances on startup to ensure no two machines have the same ID).
* **Load Balancer:** Nginx (Round Robin) sits in front of the ID generation cluster.

**Production Scenario Handling:**

* **Clock Synchronization:** If the system clock moves backwards (NTP drift), the generator refuses to generate an ID to prevent duplicates.
* **Sequence Overflow:** If a machine generates > 4096 IDs in 1ms, it waits (sleeps) until the next millisecond.

**Why this is the best solution:**

* **No Coordination:** Once a worker starts, it generates IDs independently without checking a central DB. This is infinite horizontal scalability.
* **DB Performance:** Since IDs are time-sortable, inserting them into a B-Tree index (like MySQL/Postgres Primary Key) is efficient because we are always appending to the right side of the tree, avoiding page splits.
