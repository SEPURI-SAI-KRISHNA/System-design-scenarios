Let's say We have built real-time streaming engines, decoupled storage, and generative AI search layers.

But if the logic transforming your data inside the warehouse is silently failing, your beautiful, petabyte-scale architecture is just generating highly optimized garbage.

Now, we bring software engineering unit tests into the data warehouse. We master **Advanced dbt Testing**. We are going to build the mathematical circuit breakers that halt your pipeline *before* bad data ever reaches the presentation layer.

---

### **1. EASY: The "Silent Explosion" (Singular Tests)**

**Scenario:** You are building the backend for a collaborative rule engine. Users define data-matching rules in the Streamlit UI, and you transform the rule execution logs in dbt to calculate success rates.
**The Problem:** A bug in the upstream application briefly allows a user to save a rule with a `NULL` target column. When your dbt pipeline runs the daily `JOIN` between the `rules` table and the `execution_logs` table, it encounters the `NULL`. Instead of failing, the SQL executes a Cartesian explosion, multiplying your row count by 10,000 and silently corrupting the daily success metrics.
**Task:** Write a specific query that mathematically proves this table is healthy before the downstream dashboard is allowed to read it.

#### **Solution: Singular Tests**

A Singular Test in dbt is the exact opposite of a standard data model. Instead of writing SQL to *build* a table, you write SQL to *find the failures*.

**The Architecture:**
If the test query returns exactly `0` rows, the test passes. If it returns `1` or more rows, the test fails, and dbt throws a massive red error in your terminal.

**The Workflow:**

1. **The Code:** You create a file named `tests/assert_no_null_target_columns.sql`.
2. **The Logic:** You write the SQL to hunt for the exact anomaly:
```sql
SELECT 
    rule_id, 
    created_at
FROM {{ ref('stg_matching_rules') }}
WHERE target_column IS NULL

```


3. **The Execution:** When you run `dbt test`, dbt compiles this query and runs it against the warehouse. If it finds even a single `NULL` target column, it stops the deployment. You just prevented the Cartesian explosion.

### **2. MEDIUM: The "Generic Guardrail" (Custom Generic Tests)**

**Scenario:** In the Sentinel fraud detection engine, you have 15 different tables across the Bronze, Silver, and Gold layers that store financial transaction data.
**The Problem:** You need to ensure that `transaction_amount` is never a negative number. Writing 15 separate Singular Tests (one for every single table) is a massive violation of the DRY (Don't Repeat Yourself) principle.
**Task:** Design a reusable, parameter-driven testing function that can be applied to any column in any table with a single line of configuration.

#### **Solution: Custom Generic Tests (Test Macros)**

We combine dbt Testing with the Jinja Macros we learned on Day 51. We create a "Test Factory."

**The Architecture:**
We abstract the column name and the model name into variables.

**The Workflow:**

1. **The Macro:** You create a file in the `macros/` folder named `test_is_positive.sql`:
```jinja
{% test is_positive(model, column_name) %}

    SELECT *
    FROM {{ model }}
    WHERE {{ column_name }} < 0

{% endtest %}

```


2. **The YAML Configuration:** You never write SQL for this test again. In your `schema.yml` file, you simply attach the new generic test to the relevant columns across all 15 tables:
```yaml
models:
  - name: fct_sentinel_transactions
    columns:
      - name: transaction_amount
        tests:
          - is_positive

```



**Production Grade Architecture:**
dbt ships with four native generic tests (`unique`, `not_null`, `accepted_values`, `relationships`). By writing your own custom generic tests, you can enforce massive, company-wide business logic—like *"ensure end_date is always greater than start_date"* or *"ensure this JSON string is valid"*—across thousands of models with just a few lines of YAML.

### **3. HARD: The "Circuit Breaker" (Severity & Multi-Column Testing)**

**Scenario:** You are building the final `gold_fraud_metrics` table.
**The Problem:** You discover a complex business anomaly: A transaction is marked as `fraud_flag = true`, but the `risk_score` is somehow less than 10. This is mathematically impossible according to the Sentinel scoring algorithm.
**The Disaster:** If 5 rows out of 10 million have this anomaly, it might just be a weird edge case. If 50,000 rows have it, the upstream machine learning model is completely broken. If you configure dbt to instantly crash the pipeline on the very first failure, you will wake up at 3:00 AM every single night for a 5-row edge case.
**Task:** Architect a multi-column test that dynamically triggers warnings versus catastrophic pipeline failures based on strict threshold math.

#### **Solution: Multi-Column Logic & Severity Thresholds**

We write a test that evaluates the relationship *between* columns, and we configure error tolerances in the YAML.

**The Architecture:**
We use a Singular Test to write the complex multi-column logic, but we manage the response dynamically.

**The Workflow:**

1. **The Complex Test:** `tests/assert_valid_fraud_flag_ratio.sql`
```sql
SELECT 
    transaction_id,
    fraud_flag,
    risk_score
FROM {{ ref('gold_fraud_metrics') }}
WHERE fraud_flag = TRUE AND risk_score < 10

```


2. **The Threshold Configuration:** We open our `dbt_project.yml` file and explicitly configure the tolerances for this specific test:
```yaml
tests:
  +severity: warn
  +error_if: "> 500"
  +warn_if: "> 10"

```



**Why this is best:** * If the query returns 5 anomalous rows, dbt logs it, stays green, and the pipeline finishes successfully.

* If it returns 50 rows, dbt throws a `WARNING`, sends a Slack message to the Data Quality team to investigate the anomaly in the morning, but the pipeline still finishes successfully.
* If it returns 501 rows, dbt throws a fatal `ERROR`. The pipeline violently aborts. The data is quarantined, and the CEO's dashboard is blocked from updating with wildly corrupted ML predictions.

You have built a self-regulating, autonomous quality control system.
