Now we build the Nervous System of our architecture.

In a distributed system, services cannot always talk synchronously (HTTP/gRPC) because if the receiver is down or slow, the sender crashes. We need Message Queues (Brokers) to decouple them.

We master the "Big Three": Amazon SQS (Simple), RabbitMQ (Router), and Apache Kafka (Stream).

### **1. EASY: The "Forgot Password" Email Service (Async Tasks)**

**Scenario:** You have a user management service. When a user clicks "Forgot Password," you need to send an email.
**Problem:** If you use a standard HTTP call to an external email provider (like SendGrid), it might take 2-3 seconds. If SendGrid is down, your user sees an error page.
**Task:** Decouple the user request from the email sending. Ensure no emails are lost even if the Email Worker crashes.

#### **Solution: Amazon SQS (Standard Queue)**

We use **Amazon SQS (Simple Queue Service)** because it is serverless, requires zero maintenance, and scales infinitely.

**The Workflow:**

1. **User Action:** User clicks "Reset Password".
2. **Producer:** The Web Server pushes a message `{"user_id": 101, "type": "RESET_EMAIL"}` to the SQS Queue.
3. **Response:** Web Server immediately returns `200 OK` to the user ("Check your inbox"). Latency: <50ms.
4. **Consumer:** A background worker (Lambda or EC2) polls SQS, gets the message, and calls the SendGrid API.

#### **Production Grade Architecture**

In production, "sending emails" fails often (rate limits, bad addresses). We need a **Dead Letter Queue (DLQ)** pattern.

**The Architecture:**

* **Primary Queue:** `email-tasks-queue` (Visibility Timeout: 30s).
* **Dead Letter Queue:** `email-tasks-dlq`.
* **Compute:** AWS Lambda (Consumer).

**Production Configuration:**

* **Redrive Policy:** If the Lambda fails to process the message (throws an error), SQS puts it back in the queue.
* **Max Receive Count = 3:** If the message fails 3 times, SQS moves it to the **DLQ**.
* **Alerting:** An AWS CloudWatch Alarm watches the DLQ. If `ApproximateNumberOfMessagesVisible > 0`, it pages the on-call engineer to investigate why emails are failing.

**Why this is best:**

* **Maintenance:** No servers to patch.
* **Cost:** You pay per request. For low volume (email), it's nearly free.
* **Reliability:** SQS persists data across 3 Availability Zones automatically.

### **2. MEDIUM: The "Logistics" Routing System (Complex Routing)**

**Scenario:** You are building a logistics platform for a delivery company.
**Events:**

* `Order.Created` (Needs to go to **Warehouse** service).
* `Order.Cancelled` (Needs to go to **Accounting** AND **Warehouse**).
* `Order.Shipped` (Needs to go to **Notification** service).
**Task:** Design a messaging system that routes these messages to the correct services automatically without the "Producer" knowing who is listening.

#### **Solution: RabbitMQ (Topic Exchange)**

We use **RabbitMQ** because it is a "Smart Broker." It has sophisticated routing logic (Exchanges and Bindings) that Kafka lacks.

**The Workflow:**

1. **Exchange:** We create a Topic Exchange called `logistics_events`.
2. **Routing Keys:** Producers publish messages with keys like `order.us.created` or `order.eu.cancelled`.
3. **Queues & Bindings:**
* **Warehouse Queue:** Binds with pattern `order.*.created` AND `order.*.cancelled`.
* **Accounting Queue:** Binds with pattern `order.*.cancelled`.
* **Notification Queue:** Binds with pattern `order.#` (receives everything).



#### **Production Grade Architecture**

RabbitMQ is not serverless; you must manage the nodes. If a node dies, you lose queue data unless configured correctly.

**The Architecture:**

* **Cluster:** 3 RabbitMQ Nodes across 3 Availability Zones.
* **Queue Type:** **Quorum Queues** (New standard replacing Mirrored Queues).
* Quorum Queues use the **Raft Consensus Algorithm**. Data is replicated to at least 3 nodes. If the leader dies, a follower is elected immediately.


* **Load Balancer:** A TCP Load Balancer sits in front of the 3 nodes to distribute publishers.

**Why this is best:**

* **Decoupling:** The "Order Service" doesn't know "Accounting" exists. It just shouts "Order Cancelled!" and RabbitMQ ensures Accounting gets it.
* **Filtering:** Consumers only receive what they care about, saving CPU/Network.

### **3. HARD: The "Clickstream" Analytics (High Throughput)**

**Scenario:** You are tracking user behavior on a site like Netflix.
**Volume:** 1 Million events per second (Clicks, Pauses, Hovers, Plays).
**Requirement:**

1. **Real-time:** Feed a live "Trending Now" dashboard.
2. **Historical:** Dump all data into a Data Lake (S3) for machine learning later.
3. **Replay:** If the "Trending" dashboard crashes, it must be able to restart and re-read the last 10 minutes of data.

**Task:** Design the ingestion layer. *Hint: RabbitMQ will melt at this scale.*

#### **Solution: Apache Kafka (Partitioned Log)**

We use **Kafka** because it is a **Distributed Commit Log**, not a traditional queue. It is designed for massive throughput and persistence.

**The Workflow:**

1. **Topic:** `user_clicks`.
2. **Partitions:** We split the topic into 100 Partitions.
3. **Producers:** Web servers hash the `user_id` to determine which partition to write to (ensures all clicks from User A go to Partition 5).
4. **Consumer Group A (Real-time):** Reads the stream, calculates metrics, pushes to Redis.
5. **Consumer Group B (Archival):** Reads the *same* stream (at its own pace) and writes to S3.

#### **Production Grade Architecture**

Kafka is complex. A bad configuration leads to data loss or "Split Brain."

**The Architecture:**

* **Brokers:** 5+ Kafka Brokers (r5.2xlarge EC2 instances).
* **Control Plane:** **KRaft Mode** (modern Kafka without ZooKeeper) for metadata management.
* **Storage:** EBS gp3 volumes (High IOPS).

**Critical Configurations:**

* `replication.factor = 3`: Each partition exists on 3 brokers.
* `min.insync.replicas = 2`: The producer only gets an "OK" if at least 2 brokers confirm they have the data.
* `acks = all`: The strongest guarantee. The leader waits for replicas to acknowledge.
* **Retention Policy:** Set `log.retention.hours = 168` (7 days). This allows consumers to go offline for a week and still catch up.

**Why this is best:**

* **Replayability:** Unlike SQS/RabbitMQ, reading a message in Kafka does *not* delete it. Both the Dashboard and the Data Lake can read the same message.
* **Throughput:** Kafka writes sequentially to disk (Linear I/O), which is faster than random memory access in some cases.
