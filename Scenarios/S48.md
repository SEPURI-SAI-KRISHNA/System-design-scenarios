We lock down the most critical asset in your entire infrastructure: the data itself.

When you are architecting a collaborative, lightweight file system application using Streamlit for your colleagues, security isn't just about locking down the AWS VPC. If an HR manager uploads a sensitive spreadsheet, a standard engineer shouldn't be able to query its metadata or contents just because they both happen to have access to the backend database.

Now, we master **Data Privacy & Encryption**. We are going to ensure that even if a hacker (or a nosy colleague) bypasses the network, the data they find is mathematically useless to them.

---

### **1. EASY: The "Cleartext" Vulnerability (Encryption at Rest)**

**Scenario:** Your file-sharing application allows colleagues to upload PDFs and CSVs, which your backend writes directly to an AWS S3 bucket.
**The Problem:** If someone accidentally configures the S3 bucket to be public, or if a rogue AWS admin looks at the physical hard drives in the data center, they can read the files in perfect cleartext.
**Task:** Ensure the physical bytes on the hard drive are scrambled, without forcing your application code to manually encrypt and decrypt every single file.

#### **Solution: Envelope Encryption (AWS KMS)**

We delegate the cryptographic heavy lifting to the infrastructure layer using the AWS Key Management Service (KMS).

**The Architecture:**

1. **The Customer Managed Key (CMK):** We use Terraform to create a master AES-256 key in KMS. This key *never* leaves the KMS hardware security modules.
2. **The Data Key:** When your application writes a file to S3, AWS generates a unique "Data Key" specifically for that file. It uses the Data Key to encrypt the file, and then uses the CMK to encrypt the Data Key itself.
3. **The Storage:** AWS stores the encrypted file and the encrypted Data Key next to each other in S3.

**The Workflow:**
When your Streamlit app requests the file, S3 securely asks KMS to decrypt the Data Key. If your application's IAM Role (from Day 46) has explicit permission to use the CMK, KMS decrypts the Data Key, S3 decrypts the file, and hands it to your app in cleartext.
*Why this is best:* The encryption is completely transparent to your Python code. You just call `boto3.client('s3').get_object()`, but the data is physically bulletproof at rest.

### **2. MEDIUM: The "Nosy Colleague" (Row-Level Security)**

**Scenario:** You store all the metadata for the file system (File Name, Owner ID, File Size, Department) in a single ClickHouse or PostgreSQL table.
**The Problem:** You have 50 colleagues querying this database via the Streamlit UI. If you just write `SELECT * FROM file_metadata`, everyone sees everyone else's private files.
**The Disaster:** You try to fix this in the backend Python code by adding `WHERE owner_id = current_user`. But a junior developer writes a new API endpoint and forgets to add that `WHERE` clause. Data instantly leaks.
**Task:** Architect a system where data access rules are mathematically enforced at the lowest possible level, making application-layer bugs irrelevant.

#### **Solution: Row-Level Security (RLS)**

We push the access control out of the application code and directly into the database engine.

**The Architecture:**
Modern databases (Postgres, Snowflake, ClickHouse) allow you to attach security policies directly to the tables themselves.

**The Workflow (PostgreSQL Example):**

1. **Enable RLS:** You run `ALTER TABLE file_metadata ENABLE ROW LEVEL SECURITY;`
2. **The Policy:** You create a strict rule that the database engine will evaluate on every single query:
```sql
CREATE POLICY user_isolation_policy ON file_metadata
USING (owner_id = current_user);

```


3. **The Execution:** When Colleague A logs into the database and runs `SELECT * FROM file_metadata`, the database engine intercepts the query. It silently rewrites it on the fly, forcing the `owner_id` check. Colleague A only gets 5 rows back. When Colleague B runs the *exact same query*, they get 12 different rows back.

**Production Grade Architecture:**
By enforcing this at the database layer, it doesn't matter if an engineer forgets a `WHERE` clause in the Streamlit backend, or if a data scientist connects to the database using a raw SQL client like DBeaver. The security policy is absolute and inescapable.

### **3. HARD: The "Analytical Leak" (Dynamic Masking & Tokenization)**

**Scenario:** The Data Engineering team needs to pull the `file_metadata` table into the global data warehouse (the Medallion Architecture from Day 31) to run analytics on company-wide storage usage.
**The Problem:** The metadata contains Personally Identifiable Information (PII), like the `author_email` and `document_title` (which might be "Salary_Adjustments_Alice.xlsx").
**The Disaster:** If you copy this data into the Silver layer, every data analyst in the company can see Alice's salary document. But if you delete the `author_email` column entirely, the analysts cannot track "Storage Used by User" over time.
**Task:** Give the analytical teams the statistical value of the data without exposing the actual sensitive values.

#### **Solution: Cryptographic Hashing and Dynamic Data Masking**

We must sanitize the data as it flows into the analytical environment.

**The Architecture (The Ingestion Pipeline):**
When your Airflow DAG extracts the data from the operational database to move it to the Data Lake, we apply transformations to the PII columns.

1. **One-Way Hashing (SHA-256):** * We run `author_email` through a cryptographic hash function *with a salt* (a secret string appended to the data to prevent rainbow-table attacks).
* `alice@company.com` becomes `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`.
* *Result:* The analysts can still `GROUP BY author_email` to see that "User E3B0C..." uses 50GB of storage, preserving referential integrity, but they have absolutely no idea who that user is.



**Production Grade Architecture (Dynamic Masking):**
Sometimes, executives *do* need to see the real emails in the dashboard, but standard analysts shouldn't. Instead of physically hashing the data on disk, we use **Dynamic Data Masking** in engines like Snowflake or ClickHouse.

* The data is stored in cleartext (encrypted at rest by KMS).
* You create a masking policy: *"If the querying user's role is `Executive`, show the email. If the role is `Analyst`, return `***@company.com`."*
* The database engine dynamically redacts the data in RAM at the exact millisecond the query is executed, serving different results to different people from the exact same table.
