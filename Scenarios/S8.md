For modern Engineering and System Design, the answer is almost always Kubernetes (K8s). It is the operating system of the cloud. Now, we move from "running a script" to "orchestrating a fleet."

We will cover the three pillars of K8s: Stateless Deployments, Networking (Services), and Stateful Applications.

### **1. EASY: The "Zero-Downtime" Web App (Deployments)**

**Scenario:** You have a Python Data API. You need to release Version 2.0.
**Problem:** If you stop the server to upgrade it, users get a 503 Error. You cannot have any downtime.
**Task:** Design a deployment strategy in Kubernetes that updates the app while traffic is still flowing.

#### **Solution: K8s Deployment Object (Rolling Update)**

We use the **Deployment** resource. It manages a set of identical Pods (containers).

**The Configuration:**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: data-api
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1  # Only kill 1 pod at a time
      maxSurge: 1        # Create 1 new pod above the limit
  template:
    ...

```

**The Workflow:**

1. **Trigger:** You apply the YAML with the new image `data-api:v2`.
2. **Surge:** K8s creates **one** new V2 Pod. Now you have 4 Pods running (3 V1, 1 V2).
3. **Health Check:** K8s waits for the V2 Pod to be "Ready."
4. **Teardown:** Once V2 is ready, K8s kills **one** V1 Pod.
5. **Repeat:** It repeats this until all pods are V2.

#### **Production Grade Architecture**

Simply running a Deployment isn't safe. You need **Probes** and **Limits**.

**1. Liveness & Readiness Probes:**

* **Liveness Probe:** "Is the app dead?" (e.g., Deadlock). If this fails, K8s *restarts* the container.
* **Readiness Probe:** "Is the app ready to serve traffic?" (e.g., loading ML models into RAM). If this fails, K8s *stops sending traffic* to this pod (removes it from the Load Balancer) but does not kill it.

**2. Resource Requests & Limits (The "Noisy Neighbor" fix):**

* **Request:** "I need at least 1GB RAM." (K8s uses this for scheduling).
* **Limit:** "Kill me if I use more than 2GB." (Prevents a memory leak in one app from crashing the whole node).

### **2. MEDIUM: The "Service Discovery" Puzzle (Services)**

**Scenario:**

* **Frontend Pods:** Ephemeral. They scale up and down. IPs change every hour.
* **Backend Pods:** Ephemeral. IPs change every hour.
**Problem:** How does the Frontend know the IP address of the Backend if the IPs keep changing? You cannot hardcode `10.0.0.5`.
**Task:** Implement a stable networking layer.

#### **Solution: K8s Service (ClusterIP)**

We create a **Service** object. This provides a stable Virtual IP (VIP) and a DNS entry.

**The Architecture:**

* **Labeling:** We tag Backend pods with `app: backend`.
* **Service Definition:**
```yaml
kind: Service
metadata:
  name: backend-svc
spec:
  selector:
    app: backend  # Finds any pod with this label
  ports:
    - port: 80

```


* **DNS Resolution:** Kubernetes runs an internal DNS (CoreDNS). The Frontend code simply calls `http://backend-svc`.
* **Routing:** Kube-Proxy intercepts this call and load-balances it (Round-Robin) across the healthy Backend pods.

#### **Production Grade Architecture**

**Network Policies (Internal Firewall):**
By default, any pod in K8s can talk to any other pod. This is a security risk. If a hacker compromises your generic "Web Scraper," they can talk to your "Database."

**The Fix:** Implement a **NetworkPolicy**.

```yaml
kind: NetworkPolicy
spec:
  podSelector:
    matchLabels:
      app: database
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: backend-api  # Only Backend API can touch the DB

```

* This explicitly **blocks** the Web Scraper (or anything else) from connecting to the Database port.

### **3. HARD: The "Database" in Containers (StatefulSets)**

**Scenario:** You want to run a Cassandra or Kafka cluster inside Kubernetes.
**The Problem:** Standard Deployments treat pods as interchangeable.

* If `Pod-1` dies, `Pod-4` replaces it.
* But for a Database, identity matters! `Pod-0` might be the "Master." `Pod-1` owns the data on Disk A. If `Pod-1` dies, the replacement **MUST** attach to Disk A and identify itself as `Pod-1`.
**Task:** Deploy a stateful distributed system on K8s.

#### **Solution: StatefulSet + PersistentVolumes (PV)**

We use a **StatefulSet** instead of a Deployment.

**Key Differences:**

1. **Stable Identity:** Pods are named `cassandra-0`, `cassandra-1`, `cassandra-2` (not random hashes).
2. **Ordered Deployment:** `cassandra-1` will not start until `cassandra-0` is Running and Ready.
3. **Sticky Storage:** We use a `volumeClaimTemplate`.
* K8s creates a Persistent Volume (e.g., AWS EBS Drive) for *each* pod.
* `cassandra-0` gets `disk-0`.
* If `cassandra-0` dies and restarts on a different node, K8s automatically detaches `disk-0` from the old node and re-attaches it to the new node.



#### **Production Grade Architecture**

**Storage Classes & Headless Services:**

* **StorageClass:** We define *fast* storage for DBs.
```yaml
kind: StorageClass
name: fast-ssd
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3  # AWS General Purpose SSD
  iopsPerGB: "10"

```


* **Headless Service:** Standard Services load balance (Round Robin). Database clients often need to talk to a *specific* node (e.g., "Write to the Master").
* We set `clusterIP: None` in the Service.
* This allows DNS to resolve `cassandra-0.db-svc` directly to the IP of the first pod, bypassing the Load Balancer. This is critical for internal cluster communication (Gossip Protocol).
