### **1. EASY: The "Stateless" Login (Authentication & JWTs)**

**Scenario:** You have a web application with 5 million daily active users.
**Problem:** In the old days, when a user logged in, the server created a "Session" in memory or in a database (like Redis). On every single API request, the server had to check the database: "Is session ID `xyz` valid?" At 100,000 requests per second, this database lookup becomes a massive, unnecessary bottleneck.
**Task:** Design a way for users to prove who they are *without* the backend needing to query a database on every request.

#### **Solution: JSON Web Tokens (JWT)**

We use stateless authentication. The server issues a cryptographically signed token that the client holds.

**The Workflow:**

1. **Login:** User sends `POST /login` with username/password.
2. **Verification:** Server checks the Postgres database once. It matches!
3. **Signing:** The server generates a JSON object `{"user_id": 123, "role": "admin"}` and signs it using a secret key (HMAC) or a private/public key pair (RSA). This creates the JWT.
4. **Usage:** The client saves this JWT. For every subsequent request, it attaches it: `Authorization: Bearer <JWT>`.
5. **Validation:** The receiving microservice looks at the token, hashes the payload with its known secret key, and if the signatures match, it *knows* the token is valid. Zero database lookups required.

#### **Production Grade Architecture**

**The Vulnerability:** If a hacker steals a JWT (via a malicious browser extension), they *are* that user until the token expires. You cannot easily "delete" a stateless token from the server.
**The Fix: The Token Pair Pattern.**

* **Access Token:** Short-lived (Expires in 15 minutes). Sent as a Bearer token.
* **Refresh Token:** Long-lived (Expires in 7 days). Stored in a strict **HttpOnly, Secure Cookie**. JavaScript cannot read this cookie, making it immune to Cross-Site Scripting (XSS) attacks.
* When the Access Token expires, the frontend silently hits a `/refresh` endpoint. The server reads the secure cookie, verifies it against a database (this is the *only* time we check the DB), and issues a new 15-minute Access Token.

### **2. MEDIUM: The "Third-Party" Integration (OAuth 2.0 & OIDC)**

**Scenario:** You are building a SaaS platform. You want users to "Log in with Google." Furthermore, you want to allow a third-party BI tool (like Tableau) to read the user's data from your API.
**Constraint:** The user must *never* give Tableau their password to your platform.
**Task:** Design a delegated authorization system.

#### **Solution: OAuth 2.0 (Authorization Code Flow) + OIDC**

We implement OAuth 2.0 for authorization (granting access) and OpenID Connect (OIDC) for authentication (getting the user's profile).

**The Workflow:**

1. **The Redirect:** User clicks "Connect Tableau." Tableau redirects the user's browser to *your* authorization server (`auth.yourcompany.com`).
2. **The Consent:** The user logs in to your site and sees: *"Tableau wants to read your data. Allow?"*
3. **The Auth Code:** User clicks "Allow." Your server redirects the user back to Tableau with a temporary, one-time-use **Authorization Code**.
4. **The Exchange (Back-channel):** Tableau's backend server secretly contacts your backend server, providing the Auth Code and its Client Secret.
5. **The Tokens:** Your server validates the code and responds with an Access Token. Tableau now uses this token to call your API.

#### **Production Grade Architecture**

**API Gateway Centralization:**
You have 50 microservices. If you make every single Python, Go, and Java microservice implement OAuth token validation, you will have a security nightmare when a library needs patching.

**The Fix:** Offload token validation to the Edge using an **API Gateway** (e.g., Kong, AWS API Gateway, or Envoy).

1. External request hits the API Gateway with a JWT.
2. The Gateway fetches the Public Keys (JWKS) from your Identity Provider (Okta/Auth0/Keycloak).
3. The Gateway verifies the signature and expiration.
4. The Gateway strips the JWT out of the request, extracts the `user_id`, and passes the request to the internal VPC with a spoof-proof header: `X-Internal-User-Id: 123`.
5. Internal microservices blindly trust the `X-Internal-User-Id` header because the firewall prevents outside traffic from reaching them directly.

### **3. HARD: The "Zero Trust" Data Center (mTLS & Service Mesh)**

**Scenario:** You are the architect for a FinTech app processing credit cards.
**The Threat:** The traditional security model is the "Castle and Moat." The firewall (moat) keeps bad guys out. But what if a hacker finds a vulnerability in your frontend UI pod and gets *inside* the K8s cluster?
**The Problem:** Once inside, the hacker can sniff the network traffic between the `Billing Service` and the `Postgres Database` because internal VPC traffic is usually plain-text HTTP.
**Task:** Secure internal service-to-service communication. You must assume the network is already compromised.

#### **Solution: Mutual TLS (mTLS)**

Standard TLS (HTTPS) is one-way: Your browser verifies the Bank's certificate, but the Bank doesn't verify your browser's certificate.
**Mutual TLS** means *both* sides verify each other.

* Service A (Billing) presents its cryptographic certificate to Service B (Ledger).
* Service B presents its certificate to Service A.
* If both are valid, they form a heavily encrypted, authenticated tunnel.

#### **Production Grade Architecture**

**The Service Mesh (Istio):**
Manually generating, distributing, and rotating TLS certificates for 1,000 ephemeral Kubernetes pods is humanly impossible. We use a Service Mesh.

**The Architecture:**

* **Control Plane (Istiod):** Acts as the internal Certificate Authority (CA).
* **Data Plane (Envoy Sidecars):** Every single pod gets a tiny proxy container injected next to it.
* **The Magic:** 1. The Python developer writes standard, plain-text HTTP calls: `requests.get("http://ledger-svc")`.
2. The local Envoy sidecar intercepts this plain-text call.
3. Envoy requests a short-lived certificate (e.g., expires in 1 hour) from Istiod.
4. Envoy wraps the payload in mTLS and sends it across the network to the receiving pod's Envoy.
5. The receiving Envoy decrypts it, verifies the identity, and hands plain-text HTTP to the receiving app.

**Why this is the best solution:**

* **Zero Code Changes:** Developers don't manage certificates or write TLS code.
* **Blast Radius:** If a certificate is stolen, it expires in 1 hour anyway.
* **Identity-Based Firewall:** You can write K8s policies that say: *"Only pods with a certificate cryptographically proving they are the `Billing-Service` can talk to the database."*
