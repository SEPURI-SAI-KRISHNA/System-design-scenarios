### **1. EASY: The Viral Product Launch (Read Scalability)**

**Scenario:** You are designing the backend for an e-commerce giant (like Amazon) for "Prime Day." You have a product page (e.g., iPhone 15) that is static (description, images don't change often) but will receive 1 million read requests per second. The price might change once every few hours.
**Task:** Protect your primary database from dying under the read load. Ensure the price updates are reflected reasonably quickly.

**Solution Architecture:**

* **Pattern:** **Cache-Aside Pattern (Lazy Loading)**.
* **Database:** PostgreSQL (Source of Truth).
* **Cache:** Redis Cluster (In-memory store).
* **CDN:** Cloudflare (for images/CSS).

**The Workflow:**

1. **Request:** The Application Service receives a request for `GET /product/iphone-15`.
2. **Check Cache:** The service checks Redis for key `product:iphone-15`.
3. **Hit:** If found, return data immediately (Latency: <5ms).
4. **Miss:** If not found:
* Query PostgreSQL (Latency: 20-100ms).
* Write the result to Redis with a **TTL (Time To Live)** of 10 minutes.
* Return data to user.


5. **Update Strategy:** When the Price Service updates the price in Postgres, it must also **invalidate (delete)** the key in Redis immediately. The next read will result in a "Miss" and fetch the new price.

**Why this is the best solution:**

* **Cost:** RAM (Redis) is expensive. We only cache data that is actually requested (Lazy Loading).
* **Resilience:** If Redis crashes, the system falls back to the DB (though performance drops, the system stays up).
* **Thundering Herd Problem (Production Note):** If the cache expires and 10,000 users hit the endpoint simultaneously, they all hit the DB. *Fix:* Add "Jitter" (random variance) to the TTL so keys don't expire at the exact same millisecond.

### **2. MEDIUM: The Smart City Power Grid (Write Heavy)**

**Scenario:** You are building a telemetry system for a Smart City. There are 10 million smart meters. Each meter sends a voltage reading every 15 seconds.
**Volume:** ~660,000 writes per second.
**Task:** Design the storage layer to handle this write throughput. You need to be able to query: *"Show me the voltage history of Meter X for the last 24 hours."*

**Solution Architecture:**

* **Ingestion:** Kafka (to buffer the massive write bursts).
* **Database:** **Apache Cassandra** (or ScyllaDB for C++ performance).
* **Pattern:** Wide-Column Store / Log Structured Merge (LSM) Trees.

**The Workflow:**

1. **Write Path:** Meters $\rightarrow$ Kafka $\rightarrow$ Spark Streaming (ETL/Enrichment) $\rightarrow$ Cassandra.
2. **Why Cassandra?** Relational DBs (B-Trees) require disk seeks to update indexes on every insert, which kills performance at this scale. Cassandra uses **LSM Trees** (Log Structured Merge Trees). It writes to memory (MemTable) and then sequentially flushes to disk (SSTable). Sequential writes are huge for HDD/SSD performance.

**The Data Model (Crucial Step):**
In NoSQL, we model for the *query*, not the data relationships.

```sql
CREATE TABLE meter_readings (
    meter_id uuid,
    day date,
    timestamp timestamp,
    voltage float,
    PRIMARY KEY ((meter_id, day), timestamp)
) WITH CLUSTERING ORDER BY (timestamp DESC);

```

* **Partition Key `(meter_id, day)`:** This ensures that all readings for Meter X on Day Y live on the **same physical node** on the disk.
* **Clustering Key `timestamp`:** This sorts the data on the disk by time.

**Why this is the best solution:**

* **Write Speed:** O(1) write complexity. It's essentially appending to a file.
* **Read Speed:** When you ask for "Meter X, Yesterday," the DB goes to *one* partition and reads sequentially. No joining, no scanning the whole table.
* **Scale:** Cassandra scales linearly. Need more write capacity? Just add more nodes.

### **3. HARD: The "Uber" Double Booking (Distributed Locking)**

**Scenario:** You are designing the ride-matching engine for Uber.

* **User A** sees "Driver Mike" is available.
* **User B** sees "Driver Mike" is available.
* Both tap "Book Now" at the exact same millisecond.
**Task:** Ensure Driver Mike is assigned to only *one* user. The other user must receive a "Driver no longer available" message.
**Constraint:** You have microservices running on 50 different servers. You cannot use a simple Java `synchronized` block.

**Solution Architecture:**

* **Service:** Matching Service (Stateless Kubernetes Pods).
* **Lock Manager:** **Redis (Implementing Redlock Algorithm)**.
* **Persistence:** PostgreSQL (Serializable Isolation).

**The Workflow:**

1. **Request:** User A and User B send requests to different Matching Service instances.
2. **The Distributed Lock:**
* Both instances attempt to acquire a lock in Redis for the key `lock:driver:mike`.
* We use the command: `SET lock:driver:mike "user_A_id" NX PX 5000`.
* `NX`: Only set if key does **N**ot e**X**ist.
* `PX 5000`: Key auto-expires in 5 seconds (prevents deadlocks if the service crashes after locking).




3. **The Winner:**
* Redis is single-threaded. It processes User A's command first. It returns `OK`.
* User A's service instance now "holds the lock."


4. **The Loser:**
* Redis processes User B's command. Since the key exists, it returns `NULL`.
* User B's service returns specific error: "Driver unavailable, searching for others..."


5. **State Change:**
* User A's service updates Postgres: `UPDATE drivers SET status = 'BUSY' WHERE id = 'Mike'`.
* User A's service releases the Redis lock.



**Why this is the best solution:**

* **Speed:** Redis locking is sub-millisecond. Database locking (pessimistic locking on rows) holds open a database connection and transaction, which is heavy and hard to scale.
* **Safety:** The `NX` parameter guarantees atomicity. The `PX` (TTL) guarantees liveness (we don't block the driver forever if the server crashes).
* **Production Note:** For extreme reliability (banking), we might use **ZooKeeper** or **Etcd** instead of Redis, as they use consensus algorithms (Raft/Paxos) which are safer than Redis in network partition scenarios, though slightly slower.
