We have built the system, but now we must operate it. In a distributed environment, you cannot just SSH into a server and check logs. There are 500 servers.

Now we cover Observability (Metrics, Logs, Traces) and Reliability Patterns

### **1. EASY: The "Black Box" Server (Metrics & Dashboards)**

**Scenario:** You have a Python API service running on 20 Kubernetes pods.
**Problem:** Users report "It feels slow." You have no idea if it's CPU overload, memory leaks, or just network lag.
**Task:** Implement a monitoring stack to visualize the "Four Golden Signals" (Latency, Traffic, Errors, Saturation).

#### **Solution: Prometheus (Pull-Based) & Grafana**

We do not push metrics to a server (pushing can overload the monitoring server during traffic spikes). Instead, we use **Prometheus** to *scrape* (pull) metrics.

**The Architecture:**

* **Instrumentation:** We add the `prometheus_client` library to the Python code.
* **Exposition:** The app exposes a lightweight HTTP endpoint `/metrics` that displays plain text data:
```
# HELP http_requests_total The total number of HTTP requests.
http_requests_total{method="post", status="200"} 1024

```


* **Collection:** The Prometheus server wakes up every 15 seconds, hits `/metrics` on all 20 pods, and stores the values.
* **Visualization:** **Grafana** queries Prometheus to build charts.

#### **Production Grade Architecture**

**The "Sidecar" Pattern:**
In Kubernetes, we don't want to manually configure Prometheus for every new service.

* **ServiceMonitor:** We use the Prometheus Operator. We define a YAML rule saying "Monitor any pod with label `app: my-api`."
* **AlertManager:** Prometheus is connected to AlertManager.
* *Rule:* `IF avg(http_request_duration_seconds) > 1s FOR 5m`.
* *Action:* Send Slack notification to `#devops-alerts`.



**Why this is best:**

* **Pull Model:** If the monitoring system goes down, your app doesn't crash (it just fails to be scraped). If your app is under heavy load, it doesn't waste resources trying to push metrics out.
* **Standardization:** Prometheus is the industry standard for cloud-native metrics.

### **2. MEDIUM: The "Needle in a Haystack" (Distributed Tracing)**

**Scenario:** You are debugging an E-commerce Checkout flow.

* Service A (Checkout) calls Service B (Inventory) calls Service C (Payment).
* **The Error:** 1 out of 100 requests fails with a 500 error.
* **The Pain:** You check Service A logs—it says "Downstream Error". You check Service B—it looks fine. You check Service C—it shows a timeout. Linking these logs manually is impossible.
**Task:** Implement a system to visualize the *entire lifespan* of a single request across all microservices.

#### **Solution: OpenTelemetry & Jaeger**

We implement **Distributed Tracing**. We assign a unique `Trace ID` to the request the moment it hits the Load Balancer, and we propagate this ID to every internal service.

**The Workflow:**

1. **Ingress:** Request hits Nginx. Nginx generates `Trace-ID: abc-123`.
2. **Context Propagation:**
* Service A receives header `Trace-ID: abc-123`.
* Service A calls Service B, injecting `Trace-ID: abc-123` into the HTTP headers.


3. **Spans:**
* Service A reports: "I started at 10:00:00 and finished at 10:00:05. My ID is `abc-123`."
* Service B reports: "I started at 10:00:01 and finished at 10:00:04. My ID is `abc-123`."


4. **Visualizer:** **Jaeger** (or Zipkin) collects these spans and draws a Gantt chart showing exactly where the time was spent.

#### **Production Grade Architecture**

Tracing every single request (100% sampling) generates massive data and costs a fortune in storage.

**The Strategy: Probabilistic Sampling**

* **Head-Based Sampling:** The first service decides: "I will trace 1% of traffic." It sets a flag `Sampled: 1`. Downstream services respect this flag.
* **Tail-Based Sampling (Advanced):** We collect traces for *all* requests in memory, but we only write them to disk **IF** the request resulted in an ERROR or High Latency. This ensures we always catch the bugs without storing gigabytes of "Success" logs.

**Why this is best:**

* **MTTR (Mean Time To Recovery):** Instead of guessing which service failed, you look at the trace and see the red bar instantly on Service C.

### **3. HARD: The "Netflix" Cascading Failure (Circuit Breakers)**

**Scenario:** You are designing the Netflix Homepage.

* It calls the **Movie Service** (Critical).
* It calls the **Recommendation Service** (Non-Critical).
* **The Incident:** The Recommendation Service has a bug and becomes extremely slow (30s timeout).
* **The Disaster:** The Homepage waits 30s for recommendations. All threads on the web server get stuck waiting. The entire Homepage goes down. Users can't even play movies.
**Task:** Ensure the Homepage stays up even if the Recommendation Service dies.

#### **Solution: Circuit Breaker Pattern (Resilience4j / Istio)**

We wrap the call to the Recommendation Service in a **Circuit Breaker**.

**The State Machine:**

1. **CLOSED (Normal):** Traffic flows through. We count errors.
2. **OPEN (Tripped):** If 50% of requests fail (or timeout) in the last 10 seconds:
* **Trip the Breaker.**
* **Fail Fast:** New requests to Recommendation Service fail *immediately* (0ms latency) without making the network call.
* **Fallback:** The Homepage catches this error and displays a default list ("Popular Movies") instead of personalized recommendations.


3. **HALF-OPEN (Test):** After 30 seconds, let 1 request through.
* Success? Close the breaker (resume normal traffic).
* Fail? Open it again for another 30 seconds.



#### **Production Grade Architecture**

**Bulkhead Pattern (Isolation):**
Just a circuit breaker isn't enough. We also use **Bulkheads**.

* We partition the Web Server's thread pool.
* Thread Pool A (50 threads): Talk to Movie Service.
* Thread Pool B (10 threads): Talk to Recommendation Service.


* **Result:** If Recommendation Service hangs, it fills up Pool B. But Pool A is totally empty. Users can still watch movies; they just don't see recommendations.

**Why this is best:**

* **Graceful Degradation:** The user experience degrades slightly (no recommendations) rather than catastrophically (white screen of death).
* **Resource Protection:** Prevents "Thread Starvation" on the calling server.
