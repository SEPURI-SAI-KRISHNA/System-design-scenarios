Now, we master Advanced Rate Limiting.

As a Engineer, you build pipelines that can ingest millions of events per second. But what happens when an attacker, or just a poorly written while(True) loop from a client, points a firehose at your public API? If you don't throttle them at the edge, your internal services (Kafka, Spark, databases) will drown.

Now, we look at how giants like Stripe, Twitter, and AWS protect their infrastructure without punishing legitimate users.

### **1. EASY: The "Brute Force" Login (Fixed Window Counter)**

**Scenario:** You have a public-facing authentication API.
**The Threat:** A botnet is trying to brute-force user passwords.
**Task:** Restrict every IP address to a maximum of 10 login attempts per minute. If they exceed it, return an HTTP `429 Too Many Requests`.

#### **Solution: The Fixed Window Counter**

This is the simplest algorithm to implement, making it great for basic use cases. We use an in-memory store like Redis.

**The Workflow:**

1. **The Key:** We construct a Redis key combining the user's IP and the current minute: `login:192.168.1.5:10:05`.
2. **The Increment:** When a request hits, the API gateway executes an atomic Redis command: `INCR login:192.168.1.5:10:05`.
3. **The Check:** * If the result is $\le 10$, allow the request.
* If the result is $> 10$, block the request (HTTP 429).


4. **Cleanup:** We set a Time-To-Live (TTL) on the key to expire after 60 seconds so our Redis instance doesn't run out of memory.

#### **Production Grade Architecture**

**The Boundary Vulnerability (Why this isn't enough for scale):**
The Fixed Window has a massive flaw: the "Boundary Problem."

* A hacker sends 10 requests at `10:05:59` (allowed).
* The clock ticks to `10:06:00`. The Redis key changes.
* The hacker sends another 10 requests at `10:06:01` (allowed).
* *Result:* The hacker just bypassed your limit, executing 20 requests in a 2-second window, potentially crashing your backend if this is done across millions of IPs simultaneously.

For strict API quotas, we need a smoother algorithm.

### **2. MEDIUM: The "Noisy Neighbor" SaaS API (Token Bucket)**

**Scenario:** You are building a SaaS API (like Stripe or AWS). You have different pricing tiers:

* *Basic Tier:* 10 requests/second.
* *Pro Tier:* 100 requests/second.
**The Problem:** User A (Basic) accidentally creates an infinite loop in their code and blasts your API with 5,000 requests/second. You must instantly throttle User A without dropping a single request from User B (Pro).
**Task:** Design a rate limiter that enforces strict quotas but allows for brief, legitimate bursts of traffic.

#### **Solution: The Token Bucket Algorithm**

This is the industry standard (used by Amazon and Stripe). Imagine a physical bucket that holds "tokens." Every API request costs exactly one token.

**The Logic:**

1. **Bucket Capacity:** The bucket can hold a maximum of 10 tokens (for a Basic user).
2. **Refill Rate:** A background process drops 10 new tokens into the bucket every second. If the bucket is full, new tokens spill over and are discarded.
3. **The Request:** When a request arrives, we check the bucket.
* If there is a token: We remove 1 token and process the request.
* If the bucket is empty: We drop the request (HTTP 429).



#### **Production Grade Architecture**

**Redis Lua Scripting (Avoiding Race Conditions):**
We do not actually run a background thread to "refill" millions of buckets every second. That would melt our CPU. We calculate the refill *mathematically* at the exact moment a request arrives.

**The Workflow:**
When the API Gateway receives a request, it runs a single, atomic **Lua Script** inside Redis:

1. It reads the `last_refill_time` and `current_tokens` from Redis.
2. It calculates how many tokens *should* have been added since the last request:

$$\text{New Tokens} = (\text{Current Time} - \text{Last Refill Time}) \times \text{Refill Rate}$$


3. It updates the bucket and processes the deduction.
4. *Why Lua?* Because Redis executes Lua scripts atomically. If 50 requests arrive at the exact same millisecond, the Lua script ensures they queue up and don't overwrite each other's token math (preventing race conditions).

### **3. HARD: The "Global Botnet" (Distributed Sliding Window)**

**Scenario:** You operate a massive global API.

* You have data centers in US-East, Europe, and Asia.
* A sophisticated botnet is attacking you. It knows your rate limit is 1,000 requests/minute per API key.
**The Threat:** The botnet sends 1,000 requests to US-East, 1,000 to Europe, and 1,000 to Asia simultaneously.
**The Problem:** If US-East has its own isolated Redis instance, it thinks the user is perfectly within their limits. The user just successfully executed 3,000 requests.
**Task:** Enforce a strict, mathematically perfect global rate limit.

#### **Solution: Sliding Window Log / Counter at the Edge**

We need an algorithm that solves the "Boundary Problem" of the Fixed Window, and we need to synchronize it globally. We use the **Sliding Window Counter**.

**The Algorithm (Sliding Window Counter):**
Instead of just looking at the current minute, we look at a mathematically weighted average of the *current* minute and the *previous* minute.

* If it's `10:05:15` (25% into the current minute).
* We take 75% of the requests from `10:04` and add them to the requests from `10:05`.
* This smooths out the traffic curve perfectly and stops burst attacks.

#### **Production Grade Architecture**

**Eventual Consistency at the Edge (Envoy/Redis CRDTs):**
We cannot make US-East synchronously check a database in Europe before allowing an API request. The speed-of-light delay (100ms+) would ruin API performance.

**The Distributed Architecture:**

1. **Edge Proxies (Envoy):** We push the rate limiting out of our application code and directly into our Edge API Gateways (Envoy Proxies).
2. **Local Redis Clusters:** Each region (US, EU, Asia) has its own ultra-fast local Redis cluster handling the Token Bucket/Sliding Window math in <1ms.
3. **Asynchronous Synchronization:** The regional Redis clusters are linked using **CRDTs** (Conflict-Free Replicated Data Types) or asynchronous gossip protocols (like Redis Enterprise Active-Active).
4. **The Catch-Up:** US-East allows the first 1,000 requests. A few milliseconds later, the gossip protocol tells Europe and Asia: *"Hey, User 123 just burned their quota in the US."* If the botnet tries to hit Europe, Europe's local Redis already knows the global bucket is empty and blocks them.

**Why this is best:** You might accidentally allow 1,050 requests instead of exactly 1,000 due to the milliseconds it takes for the regions to sync, but you maintain lightning-fast $O(1)$ latency for legitimate users while effectively neutralizing the global DDoS attack.
