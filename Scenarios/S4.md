As a Engineer, you might be used to JDBC or ODBC connections. In System Design, however, the way services talk to each other (Synchronous vs. Asynchronous, Text vs. Binary) defines the latency and reliability of your entire system.

### **1. EASY: The "Chatty" Mobile App (API Gateway & GraphQL)**

**Scenario:** You are designing the backend for a mobile app like Instagram.
**The Problem:** To render the "Profile Page," the mobile app currently has to make 3 separate REST API calls:

1. `GET /users/123` (fetches name, bio)
2. `GET /users/123/posts` (fetches last 10 images)
3. `GET /users/123/followers` (fetches count)
**Impact:** This causes "Head-of-Line Blocking" on the mobile network, drains battery, and feels slow on 4G networks.
**Task:** Redesign the API to fetch exactly what is needed in a single request.

#### **Solution: GraphQL (Apollo Federation)**

We replace the multiple REST endpoints with a single **GraphQL** endpoint. The client specifies exactly the data structure it needs.

**The Query:**

```graphql
query {
  user(id: "123") {
    name
    bio
    posts(limit: 10) {
      imageUrl
      likes
    }
    followerCount
  }
}

```

#### **Production Grade Architecture**

We don't just put a GraphQL server in front of a database. We use the **BFF Pattern (Backend For Frontend)**.

**The Architecture:**

* **Client:** Mobile App.
* **Edge Layer:** **Apollo Router / Gateway**.
* **Microservices (Subgraphs):**
* **User Service (REST):** Owns user profile data in Postgres.
* **Content Service (gRPC):** Owns posts in Cassandra.
* **Social Graph Service (Thrift):** Owns followers in Neo4j.



**The Workflow:**

1. **Request:** App sends the GraphQL query to the **Apollo Gateway**.
2. **Orchestration:** The Gateway parses the query plan. It knows which service owns which field.
* It calls User Service for `name, bio`.
* * Concurrently* calls Content Service for `posts`.


* * Concurrently* calls Social Service for `followerCount`.




3. **Response:** The Gateway stitches the 3 responses into one JSON and sends it back to the phone.

**Why this is best:**

* **Over-fetching Solved:** We don't download the user's address or email (which standard REST might return) if the UI doesn't need it.
* **Under-fetching Solved:** One round-trip instead of three.

### **2. MEDIUM: The High-Speed Microservices (gRPC vs. REST)**

**Scenario:** You are building an internal Recommendation Engine.

* **Service A (Frontend BFF):** Needs recommendations for a user.
* **Service B (AI Model):** Calculates recommendations.
**Constraint:** Service A calls Service B **50,000 times per second**.
**Problem:** Using JSON over HTTP/1.1 (REST) is burning 40% of your CPU just parsing strings (serialization/deserialization). The JSON payload is heavy (field names like `"user_preference_score"` are repeated 50k times).
**Task:** Cut latency and CPU usage by 50%.

#### **Solution: gRPC with Protocol Buffers**

We switch from text-based JSON to binary **Protocol Buffers (Protobuf)** over HTTP/2.

**How it works:**

1. **Definition (`.proto` file):** We define the contract strictly.
```protobuf
message RecRequest {
  int64 user_id = 1;
  repeated string recent_items = 2;
}

```


2. **Binary Transport:** Instead of sending the string `{"user_id": 12345}`, gRPC sends a tiny binary packet (e.g., `08 39 ...`). Field names are *not* sent over the network, only field IDs (1, 2).

#### **Production Grade Architecture**

**The Architecture:**

* **Transport:** HTTP/2 (allows multiplexing—multiple requests over a single TCP connection).
* **Service Discovery:** Kubernetes (CoreDNS) or Consul.
* **Load Balancing:** Client-Side Load Balancing.

**The Workflow (Client-Side LB):**

* *Standard LB problem:* If you put a standard Load Balancer (like Nginx) in front of gRPC, it breaks. Since gRPC uses persistent HTTP/2 connections, the LB might pin all requests to one server.
* *Production Fix:* We use a **Service Mesh (Istio/Linkerd)** or **gRPC Client stubs**.
1. Service A asks K8s: "Where are the Service B pods?"
2. K8s returns 5 IPs: `10.0.0.1` to `10.0.0.5`.
3. Service A opens persistent connections to *all 5* IPs.
4. Service A sends Request 1 to IP 1, Request 2 to IP 2 (Round Robin *inside* the client).



**Why this is best:**

* **Performance:** Protobuf is 7-10x faster to serialize/deserialize than JSON.
* **Payload Size:** 30-50% smaller payloads means less network bandwidth.
* **Strict Typing:** The `.proto` file enforces types, preventing "String vs Int" bugs between teams.

### **3. HARD: The "Twitter" Global Traffic Router (L4 vs L7 Load Balancing)**

**Scenario:** You are the infrastructure architect for a global platform.

* **Traffic:** 10 Million Concurrent Connections (WebSockets + HTTPS).
* **Requirements:**
* SSL Termination (Decrypting HTTPS) cannot happen on the application servers (too CPU intensive).
* `/api/video` requests must go to the "Video Cluster".
* `/api/chat` requests must go to the "Chat Cluster".
* Block DDoS attacks.



**Task:** Design the edge entry point for this system.

#### **Solution: Multi-Tier Load Balancing (The Onion Approach)**

A single Nginx server cannot handle 10M connections. We need layers.

**The Architecture Layers:**

1. **Layer 1: DNS & Anycast (Cloudflare / AWS Route53)**
* Users hit `twitter.com`. DNS resolves to the *closest* physical data center IP (Anycast routing). If a user is in London, they hit the London Edge PoP (Point of Presence).


2. **Layer 2: L4 Load Balancer (IPVS / HAProxy / AWS NLB)**
* **Job:** Pure packet routing. It looks at TCP/UDP headers (IP + Port). It does *not* look at the URL.
* **Performance:** Extremely fast. Can handle millions of packets per second.
* **Action:** Distributes packets to a fleet of L7 Proxies.


3. **Layer 3: L7 Load Balancer (Nginx / Envoy / AWS ALB)**
* **Job:** "Smart" Routing. It terminates SSL (decrypts the packet). It reads the HTTP headers and URL path.
* **Logic:**
* If path starts with `/api/video` $\rightarrow$ Route to Video Service.
* If path starts with `/api/chat` $\rightarrow$ Route to Chat Service.


* **Security:** Checks for SQL Injection or malicious headers (WAF).



#### **Production Grade Configuration**

**The "Service Mesh" Integration:**
Once the request passes the L7 LB and enters the Kubernetes cluster, we don't let it swim freely. We use **Istio (Envoy Proxy)** as a sidecar.

* **Ingress Gateway (Envoy):** Receives traffic from L7 LB.
* **Sidecar Proxy:** Every microservice pod has a tiny proxy helper running next to it.
* **mTLS (Mutual TLS):** The Ingress encrypts traffic *again* before sending it to the internal service. This ensures "Zero Trust" security (even if a hacker gets inside the cluster, they can't sniff traffic between services).

**Why this is best:**

* **Specialization:** L4 is optimized for raw throughput. L7 is optimized for logic and security. Separation of concerns.
* **Scalability:** We can scale the L7 layer (add more Nginx servers) independently of the L4 layer.
* **Security:** SSL termination at the edge protects internal app servers from spending CPU on encryption/decryption.
