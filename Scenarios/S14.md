When you build social media, a lost "Like" is a minor bug. When you build e-commerce, a dropped request or a double-processed event means you just stole $1,000 from a customer, or you sold an iPhone you don't actually have in the warehouse.

Now, we master High-Concurrency Inventory, Distributed Carts, and Idempotency.

### **1. EASY: The "Never-Lose" Shopping Cart (High-Availability Storage)**

**Scenario:** You are building the shopping cart for Amazon.
**The Problem:** Millions of users are adding and removing items constantly. If you use a traditional relational database (Postgres) and do a SQL `UPDATE` every time someone clicks "Add to Cart", the database will lock up under the massive write-heavy load. However, if a user adds an item on their phone, it *must* appear on their laptop. You cannot lose their cart, ever.
**Task:** Design a highly available, extremely fast storage layer for the Shopping Cart.

#### **Solution: Amazon DynamoDB (NoSQL Key-Value Store)**

We decouple the cart from the main transactional order database. We use **DynamoDB** (or Cassandra) because it provides single-digit millisecond latency at any scale.

**The Workflow:**

1. **The Schema:** We don't use relational tables. We use a single Key-Value document.
* **Partition Key:** `user_id`
* **Value:** A JSON blob containing the cart items: `{"items": [{"id": 101, "qty": 2}], "last_updated": 1715000000}`


2. **The Write:** When the user clicks "Add", the Cart Microservice fetches the JSON blob, appends the new item in memory, and overwrites the document in DynamoDB.
3. **The Read:** When the user opens the cart page, it's a lightning-fast $O(1)$ lookup by `user_id`.

#### **Production Grade Architecture**

**Handling Abandoned Carts (Cost Control):**
Storing millions of abandoned carts forever will cost a fortune in DynamoDB storage fees.

* **DynamoDB TTL (Time to Live):** We add an `expires_at` Unix timestamp attribute to every cart (e.g., `current_time + 30 days`).
* **Automated Cleanup:** DynamoDB automatically runs a background process that deletes any row where `expires_at` is in the past. This costs $0 in read/write capacity and requires zero cron jobs on your end.
* **Analytics Event:** Before DynamoDB deletes it, we can configure **DynamoDB Streams** to catch the deletion event and push it to a Kafka topic. An analytics worker reads this and sends the user a "You left something in your cart!" email.

### **2. MEDIUM: The "Flash Sale" (High-Concurrency Inventory)**

**Scenario:** It is Black Friday. You are selling the new PS5 for 50% off. You have exactly **500 units** in the warehouse.
**The Problem:** At exactly midnight, 100,000 users click "Buy Now" at the exact same millisecond. If 1,000 users concurrently query the database (`SELECT stock FROM inventory WHERE item='PS5'`), they will all see `stock = 500`. They all pass the validation, and the system accidentally sells 1,000 units. You just oversold by 500.
**Task:** Design an inventory reservation system that handles extreme concurrency without locking up the database or overselling a single unit.

#### **Solution: Redis Lua Scripting (Atomic Operations)**

Standard database transactions (Pessimistic Locking) will cause 99,500 users to experience a database timeout while waiting in line. We need to handle the math in RAM.

**The Workflow:**

1. **Pre-warming:** Before the sale starts, we load the inventory into **Redis**: `SET inventory:ps5 500`.
2. **The Request:** 100,000 requests hit the Inventory Microservice.
3. **The Atomic Decrement:** We execute a **Lua Script** inside Redis.
* *Why Lua?* Redis is single-threaded and guarantees that a Lua script executes *atomically*. No other command can interrupt it.
* *The Logic:* ```lua
local stock = tonumber(redis.call("GET", KEYS[1]))
if stock > 0 then
redis.call("DECR", KEYS[1])
return 1 -- Success
else
return 0 -- Sold Out
end
```

```




4. **The Result:** The first 500 requests get a `1` (Proceed to checkout). The remaining 99,500 requests instantly get a `0` (Show "Out of Stock" UI).

#### **Production Grade Architecture**

**The Reservation Pattern (Saga Preparation):**
Getting a `1` from Redis means you *reserved* the item, not that you bought it.

* **The Timeout:** When Redis decrements the stock, we also write a key to Redis: `SET reservation:ps5:user123 "pending" EX 600` (Expires in 10 minutes).
* **Checkout Queue:** The user is placed into a Kafka queue to process their payment.
* **The Reversal:** If the user abandons checkout or their credit card declines, a background worker sees the reservation expire, and increments the stock back up: `INCR inventory:ps5`, allowing someone else to buy it.

### **3. HARD: The "Double Spend" (Payment Idempotency)**

**Scenario:** The user is at the final checkout step. They click "Pay $1,000".
**The Threat:** The request reaches your Payment Gateway. Your server talks to Stripe, charges the $1,000, but right before your server can reply "Success" to the user, the user's Wi-Fi drops.
**The Disaster:** The user's screen says "Network Error". Panicking, they click "Pay $1,000" again. If your system is naive, it processes the new request and charges them a second time.
**Task:** Design a mathematically flawless Payment API that guarantees a user is charged exactly once, no matter how many times they retry or how many servers crash.

#### **Solution: The Idempotency Key Pattern**

An API is "idempotent" if calling it once has the exact same effect as calling it 10,000 times. We achieve this using a unique key generated by the client.

**The Workflow:**

1. **Client Generation:** When the checkout page loads, the frontend (React/iOS) generates a UUID (e.g., `Key-A1B2`).
2. **The Request:** The user clicks Pay. The frontend sends `POST /api/charge` and includes the header: `Idempotency-Key: Key-A1B2`.
3. **The Database Check:** The Payment Microservice receives the request. It immediately checks a specialized database table (usually Postgres):
`SELECT status, response FROM idempotency_keys WHERE key = 'Key-A1B2'`
4. **The State Machine:**
* **If NOT FOUND:** This is a new request. Insert the key with status `IN_PROGRESS`. Call Stripe. Update the row to `COMPLETED` and save the Stripe receipt in the `response` column. Send to user.
* **If IN_PROGRESS:** The user double-clicked really fast. The server returns `409 Conflict` ("Please wait, transaction is processing").
* **If COMPLETED:** The user's Wi-Fi dropped and they retried. The server *does not call Stripe*. It simply reads the saved `response` from the database and returns it again: `200 OK: Here is your receipt`.



#### **Production Grade Architecture**

**Atomic Database Constraints (The Ultimate Failsafe):**
If two requests with the same `Idempotency-Key` hit two different load-balanced servers at the *exact same millisecond*, both servers might query the DB, see "NOT FOUND", and both charge the card.

* **Unique Index Constraint:** We rely on the absolute ACID guarantees of the relational database. We put a `UNIQUE` constraint on the `idempotency_key` column.
* **The Race Condition Fix:** When the servers try to `INSERT` the key, Server A succeeds. Server B's database driver throws a `UniqueViolation` error. Server B instantly knows it lost the race, catches the exception, and falls back to checking the status of Server A's transaction.
* **Stripe Passthrough:** We also pass this exact same `Idempotency-Key` to Stripe's API. Even if our database catches fire, Stripe has its own idempotency layer and will reject the duplicate request on their end.
