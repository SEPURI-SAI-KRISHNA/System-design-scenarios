A glaring question remains: *How do we actually get the data out of the live production databases and into Kafka or Spark without destroying the database's performance?*

Now, we solve this with **Change Data Capture (CDC)**.

As a Engineer, you know that running heavy `SELECT` queries against a live production database is a fireable offense. Today, we architect a way to silently siphon data out of operational systems in real-time, building the exact ingestion pipelines used by giants like Netflix and Uber.

---

### **1. EASY: The "Dual Write" Problem (Why CDC Exists)**

**Scenario:** You have a microservice that processes user payments.
**The Problem:** When a payment succeeds, the service must do two things:

1. `UPDATE` the user's balance in PostgreSQL.
2. Publish a `PaymentSucceeded` event to Apache Kafka for the analytics and rewards teams.
**The Disaster:** This is the infamous "Dual Write" problem. If the Postgres transaction commits, but the network connection to Kafka drops, the analytics team never gets the event. If you write to Kafka *first*, but the Postgres transaction rolls back due to a constraint violation, the analytics team thinks a payment happened that actually didn't.
**Task:** Design a system that mathematically guarantees the Kafka event is published *if and only if* the Postgres database commits.

#### **Solution: The Transactional Outbox Pattern & CDC**

We remove the responsibility of writing to Kafka from the application code entirely.

**The Workflow:**

1. **The Application:** The service opens a single database transaction. It updates the `balance` table, and it inserts a JSON record into a new table called `outbox`. It commits. Because it's a single database transaction, it is 100% atomic (both succeed, or both fail).
2. **The CDC Process:** We deploy a Change Data Capture tool (like **Debezium**). Debezium silently monitors the database.
3. **The Push:** When Debezium sees a new row hit the `outbox` table, it reads it and pushes it to Kafka.

#### **Production Grade Architecture**

**Logical Decoding (The Silent Reader):**
How does Debezium know a row was added without running a continuous, CPU-heavy `SELECT * FROM outbox` query?

* It doesn't use SQL. It acts like a database replica.
* In PostgreSQL, we enable `wal_level = logical`. Debezium connects to Postgres and reads the **Write-Ahead Log (WAL)**—the raw, binary append-only file where Postgres records changes before writing them to the actual tables.
* **Why this is best:** Reading the WAL has practically zero impact on the database's query performance. It is extremely fast, fully asynchronous, and guarantees that every single committed row is captured exactly once.

### **2. MEDIUM: The "Zero-Impact" Analytics Ingestion (Schema Evolution)**

**Scenario:** You are migrating data from a monolithic MySQL database into your Data Lake (S3) and a ClickHouse cluster for the data science team.
**Volume:** 15,000 transactions per second.
**The Problem:** The software engineering team frequently alters the MySQL schema (e.g., `ALTER TABLE orders ADD COLUMN discount INT`). If your CDC pipeline pulls a new column that ClickHouse isn't expecting, the downstream insertion jobs will crash, and data will back up in Kafka.
**Task:** Design a streaming CDC pipeline that safely handles upstream schema changes without breaking the downstream Data Engineering pipelines.

#### **Solution: Debezium to Kafka with a Schema Registry**

We cannot send raw, schemaless JSON through the pipeline at this scale. We must enforce strict data contracts.

**The Architecture:**

* **Source Connector:** Debezium MySQL Connector.
* **Message Bus:** Apache Kafka.
* **Format:** Apache Avro.
* **Contract Manager:** Confluent Schema Registry.
* **Sink Connector:** Kafka Connect (ClickHouse Sink).

**The Workflow:**

1. **The Capture:** Debezium reads the MySQL `binlog` (the MySQL equivalent of the WAL).
2. **The Schema Check:** Before sending the event to Kafka, Debezium checks the Schema Registry. "Does the registry know about this `discount` column?"
3. **The Evolution:** If configured for *Forward Compatibility*, the Schema Registry registers the new schema version (v2). Debezium serializes the data into tightly compressed Avro binary and sends it to Kafka.
4. **The Sink:** The Kafka Connect ClickHouse Sink consumes the Avro data, looks up the schema from the Registry, and intelligently dynamically alters the ClickHouse table or null-pads the missing fields according to your evolution rules.

**Production Grade Architecture**

**Handling Initial Snapshots:**
When you plug Debezium into a database for the first time, the `binlog` only contains *new* changes. It doesn't contain the 50 million rows that were inserted last year.

* **The Snapshot Phase:** Debezium handles this automatically. When it first starts, it grabs a global read lock, takes a massive snapshot of the current table state (using a highly optimized `SELECT`), streams all 50 million rows into Kafka, and *then* seamlessly transitions to reading the live `binlog`. You must configure Kafka's retention policies to handle this massive initial burst of data so the disk doesn't fill up.

### **3. HARD: The Real-Time Fraud Engine (Stateful Stream Joins)**

**Scenario:** You are building a real-time Sentinel Fraud Detection engine. You have a massive firehose of raw credit card swipes coming into Kafka at 50,000 events/sec.
**The Problem:** To evaluate if a swipe is fraudulent, your streaming engine needs the user's *trusted home location* and *account status*, which live in a live PostgreSQL database that is constantly updated by the user profile team.
**The Disaster:** If your streaming engine pauses to make a JDBC network call to Postgres for every single swipe (50,000 times a second), you will completely DDoS the production database and add 50ms of latency per event, missing your sub-200ms fraud detection SLA.
**Task:** Design a low-latency architecture to join a massive real-time event stream with a mutating relational database.

#### **Solution: Flink Broadcast State with CDC**

We bring the database *into* the stream processing engine. We use **Apache Flink** combined with Debezium CDC.

**The Architecture:**

1. **The Fast Stream:** Raw card swipes flow into a Kafka topic (`card_swipes`).
2. **The Slow Stream (CDC):** We use Debezium to capture every `UPDATE` or `INSERT` on the Postgres `user_profiles` table and stream it into a second Kafka topic (`user_updates_cdc`).

**The Workflow (Inside Flink):**

1. **Ingestion:** The Flink job consumes *both* topics simultaneously.
2. **Broadcast State:** Flink takes the `user_updates_cdc` stream and *broadcasts* it to every single worker node in the Flink cluster.
3. **Local RocksDB:** Flink saves these user profiles into its local, embedded RocksDB state. The Flink cluster now essentially holds a real-time, in-memory materialized view of the Postgres database.
4. **The Join:** When a `card_swipe` event arrives for User 123, Flink does a local RAM/SSD lookup in RocksDB to get the user's home location.
5. **The Result:** The latency is < 1 millisecond. Zero network calls are made to the live Postgres database. The fraud model executes instantly.

#### **Production Grade Architecture**

**Handling Tombstones (Memory Leaks):**
If a user deletes their account, Postgres deletes the row.

* **The Tombstone:** Debezium reads the delete from the WAL and sends a "Tombstone" message to Kafka (a message where the key is `user_id: 123` and the value is `NULL`).
* **State Cleanup:** Your Flink job *must* be explicitly programmed to listen for these `NULL` values. When it sees one, it must call `state.remove(user_id)`. If you forget this step, your Flink RocksDB state will grow infinitely with deleted users until the cluster crashes from Out Of Memory (OOM) errors.
* **Log Compaction:** The `user_updates_cdc` Kafka topic must be configured with `cleanup.policy=compact`. This ensures Kafka only keeps the *latest* state of each user, preventing new Flink workers from having to process 5 years of historical address changes when they boot up.
